# ai-toolkit 五步流程检查清单

> **用途**：任务启动时自动展示，确保严格执行 designing/SKILL.md 规范  
> **版本**：v1.0  
> **创建时间**：2026-03-18

---

## 📋 五步流程清单

### 步骤 1：启动分析
- [ ] **识别输入类型**
  - [ ] 简短描述 → 需要询问完善
  - [ ] 完整 PRD → 跳过询问
  - [ ] 现有文档 → 审查/增量模式
- [ ] **确定输出要求**
  - [ ] 一段话（快速总结）
  - [ ] 单个 PRD（小需求默认）
  - [ ] 单个 TRD
  - [ ] PRD + TRD（大需求默认）
  - [ ] 自定义格式（N 个 md 文档）
- [ ] **判断小需求**（满足多数→仅 PRD）
  - [ ] 功能点少（1-2 个简单功能）
  - [ ] 不涉及架构变更
  - [ ] 无需新技术栈引入
  - [ ] 实现方案显而易见
  - [ ] 不涉及跨模块协作
- [ ] **检测模式**
  - [ ] 新建（无文档，用户描述新项目）
  - [ ] 直出（用户输入已是完整需求/PRD）
  - [ ] 审查（有文档，用户要求审查）
  - [ ] 增量（有代码，用户添加功能）

### 步骤 2：确认（❗强制）
- [ ] **与主人确认模式**
  - [ ] 输出格式确认（PRD / TRD / PRD+TRD）
  - [ ] 示例话术："建议采用 [格式]，是否确认？"

### 步骤 3：询问需求（❗强制）
- [ ] **一次一问，先建议后提问**
  - [ ] 功能维度：正常流程 → 失败处理 → 边界情况 → 状态转换
  - [ ] 数据维度：字段 → 类型约束 → 实体关系 → 生命周期
- [ ] **终止条件检查**
  - [ ] 主人说"ok"
  - [ ] 正常 + 失败 + 边界情况都覆盖了

### 步骤 4：按要求输出
- [ ] **生成文档**
  - [ ] PRD.md（或 PRD+TRD）
- [ ] **技术选型规范**
  - [ ] 推荐前搜索当前最佳实践
  - [ ] 每个选择有比较表
  - [ ] 无已弃用/不兼容/过时方案

### 步骤 5：一致性检查（❗强制）
- [ ] **生成前自检**（全部必须为真）
  - [ ] 覆盖率 100%？（相对于选定的输出类型）
  - [ ] 所有失败路径都问了？（直出模式例外）
  - [ ] 所有字段都确认了？（直出模式例外）
  - [ ] 没有未确认的假设？
  - [ ] 没有模糊描述？
- [ ] **生成后检查**（多文档输出时）
  - [ ] 各文档术语一致
  - [ ] 需求文档 ↔ 技术文档
  - [ ] 类型引用 ↔ 类型定义
- [ ] **显式汇报检查结果**

## 🎯 完成标准（前 4 项不可协商）

| 条件 | 说明 | 可协商 |
|------|------|--------|
| 覆盖率 100% | 相对于选定输出类型的必需内容 | ❌ 不可协商 |
| 无模糊术语 | 所有输出类型都适用 | ❌ 不可协商 |
| 一致性通过 | 多文档输出时检查 | ❌ 不可协商 |
| 自检通过 | 按模式适用的检查项 | ❌ 不可协商 |
| 用户确认 | 最终确认 | ✅ 可协商 |

## ⚠️ 常见陷阱（8 条禁令）

- [ ] ❌ 假设用户需求而不确认 → ✓ 每个关键决策都要确认
- [ ] ❌ 一次问多个问题 → ✓ 一次一问，降低用户负担
- [ ] ❌ 跳过失败路径的询问 → ✓ 正常流程、失败处理、边界都要覆盖
- [ ] ❌ 未确认就直接写入文档 → ✓ 先确认再输出
- [ ] ❌ 使用模糊术语（"适当的"、"一些"） → ✓ 所有描述必须具体可测试
- [ ] ❌ PRD/TRD 术语不一致 → ✓ 输出前执行一致性检查
- [ ] ❌ PRD/TRD 职责混乱 → ✓ PRD 只包含需求+UI 设计，TRD 只包含技术 + 数据库设计
- [ ] ❌ TRD 代码细节过多 → ✓ 只包含重要的枚举、接口定义、方法签名
