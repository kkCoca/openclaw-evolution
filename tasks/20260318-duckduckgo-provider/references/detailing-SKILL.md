# 执行方案生成

根据用户输入，生成或审查 detail 文档。

## 步骤 1：检测模式

| 模式 | 触发条件 | 工作流 |
|------|---------|--------|
| **新建** | 无 detail.md 或用户要求重新生成 | 使用 [新建工作流](#新建工作流) |
| **审查** | 有 detail.md，用户要求审查 | 使用 [审查工作流](#审查工作流) |

---

## 新建工作流

复制此清单：

```
新建方案进度：
- [ ] 步骤 1：理解输入（prd/trd/roadmap/用户描述）
- [ ] 步骤 2：确认生成范围（全栈/前端/后端/部分）
- [ ] 步骤 3：阅读相关代码，理解现状
- [ ] 步骤 4：阅读最佳实践（如果有）
- [ ] 步骤 5：按模板生成执行方案
- [ ] 步骤 6：写入 detail 文档
- [ ] 步骤 7：用户评审确认
```

### 步骤 2：读取输入

prd/trd/roadmap/用户描述

### 步骤 3：确认生成范围

使用交互选择工具询问：
- 全栈（所有任务）
- 仅前端任务
- 仅后端任务
- 指定任务

### 步骤 4：阅读相关代码

1. 定位涉及的模块/文件
2. 理解代码现状和依赖关系
3. 确定需要新增/修改/删除的文件和方法

### 步骤 5：阅读最佳实践（如果有）

**实施前必须阅读对应技术栈的最佳实践文档**

### 步骤 6：生成执行方案

按模板生成

关键约束：
- 按文件组织，标注 `[新增]`/`[修改]`/`[删除]`
- 方法签名必须完整
- **禁止写实现代码**

### 步骤 7：用户评审

根据用户反馈修正，直到确认通过。

## 审查工作流

复制此清单：

```
审查方案进度：
- [ ] 步骤 1：理解输入（prd/trd/roadmap/用户描述）
- [ ] 步骤 2：阅读 CHECKLIST.md
- [ ] 步骤 3：复制 CHECKLIST 到回复
- [ ] 步骤 4：阅读最佳实践（如果有）
- [ ] 步骤 5：逐条检查并立即记录结论
- [ ] 步骤 6：输出问题汇总表
- [ ] 步骤 7：用户确认后逐一修复
```

### 步骤 2：理解输入

prd/trd/roadmap/用户描述

### 步骤 3：执行审查

**必须阅读：CHECKLIST.md**

审查原则：
- **逐条执行**：不可跳过任何检查项
- **立即记录**：每条检查后立即写下结论
- **代码验证**：每条描述必须与实际代码对照
- **自行验证**：需确认 xx 是否 xx，深入分析自行确认而不是抛给用户
- **不确定时声明**："深入分析后我仍然无法确认 xxx，因为..."

### 步骤 4：输出问题汇总

**建议修改**：给出明确的最佳实践方案，而不是抛出多个选项

```markdown
## 审查结果

| 编号 | 检查项 | 状态 | 问题描述 | 建议修改 |
|------|--------|------|---------|---------|
```

### 步骤 5：修复问题

用户确认后，逐一修复问题列表中的项。

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| ❌ 不读代码就猜测文件结构 | ✓ 先 Grep/Glob 定位，再 Read 验证 |
| ❌ 在方案中写实现代码 | ✓ 只写方法签名和说明 |
| ❌ 审查时批量标记"通过" | ✓ 逐条检查，立即记录 |
| ❌ 跳过"不确定"的检查项 | ✓ 明确声明不确定及原因 |
| ❌ 遗漏配置/数据库变更 | ✓ 检查附录是否完整 |
