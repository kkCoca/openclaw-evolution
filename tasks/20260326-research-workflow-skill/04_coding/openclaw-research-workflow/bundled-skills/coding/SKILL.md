# 编码

文档是参考，不是圣经。错了就质疑。类型安全优先。

## 工作流程

复制此清单：

```
编码进度：
- [ ] 步骤 1：检测模式（文档驱动/无文档）
- [ ] 步骤 2：审查文档（文档驱动模式必须）
- [ ] 步骤 3：阅读最佳实践（如果有）
- [ ] 步骤 4：实施（先接口/类型 → 再模块 → 再集成）
- [ ] 步骤 5：验证（对照验收标准）
- [ ] 步骤 6：单元测试（后端必须/前端可选）
- [ ] 步骤 7：代码优化（可选）
```

## 步骤 1：检测模式

扫描用户指定目录，确定模式：

| 模式       | 条件                        | 操作                  |
|----------|---------------------------|---------------------|
| **文档驱动** | prd/trd/roadmap/detail等文档 | 审查文档 → 实施           |
| **无文档**  | 没有文档                      | 建议先用 `designing` 技能 |

宣布检测到的模式后继续。

## 步骤 2：审查文档

在文档驱动模式下，审查用户提供的文档是否有问题。见 [DOC-REVIEW.md](DOC-REVIEW.md) 获取清单。

- **发现问题**：向用户展示，询问是否先修复
- **无问题**：声明"文档审查通过"，然后继续

## 步骤 3：阅读最佳实践文档

**实施前必须阅读对应技术栈的最佳实践文档**

如何自动检测技术栈，见：[STACK-CHECK-LIST.md](../.best-practices/STACK-CHECK-LIST.md)

## 步骤 4：实施

### 编码前上下文保护

1. 分析用户提供的文档，读取类型/接口定义
2. 搜索现有的类似实现
3. 确认命名匹配项目约定

如果发现重复 → 建议复用现有实现。

### 代码标准

| 规则     | 说明                                      |
|--------|-----------------------------------------|
| 类型明确   | 避免模糊类型（Java: 避免 raw type；TS: 避免 any）    |
| 先接口后实现 | 定义清晰的接口/抽象，再写具体实现                       |
| 查阅文档   | 使用不熟悉的 API 前查阅官方文档                      |
| 遵循约定   | 匹配项目现有的命名、结构、风格                         |
| 文档注释清晰 | 代码块解释、方法注释、类文件注释（署名作者统一是： by Ai.Coding） |

## 步骤 5：验证

### 验证清单

```
验证进度：
- [ ] 功能完整性：所有需求点都已实现
- [ ] 类型安全：无 any/raw type，类型定义完整
- [ ] 代码质量：无重复代码，遵循项目约定，代码符合最佳实践文档
- [ ] 边界处理：异常情况、空值、边界条件已处理
- [ ] 可测试性：关键逻辑可单元测试
```

### 验证方式

| 验证项       | 方法                      |
|-----------|-------------------------|
| **功能完整性** | 对照 prd/trd/roadmap 逐条核对 |
| **类型安全**  | 检查类型定义，确保无模糊类型          |
| **代码质量**  | 搜索相似代码，确认无重复，代码符合最佳实践文档 |
| **边界处理**  | 审查 null/undefined/空集合处理 |
| **可测试性**  | 确认核心逻辑可独立测试             |

验证通过后声明："验证完成，所有检查项通过"

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| ❌ 不读文档直接写代码 | ✓ 文档驱动模式必须先审查文档 |
| ❌ 不读最佳实践就编码 | ✓ 检测技术栈，阅读对应最佳实践 |
| ❌ 需求变更时只改代码 | ✓ 按 prd → trd → roadmap → detail → 代码顺序 |
| ❌ 重复造轮子 | ✓ 先搜索现有实现，优先复用 |
| ❌ 不熟悉 API 就猜测用法 | ✓ 查阅官方文档后再使用 |
| ❌ 忽略项目约定自创风格 | ✓ 匹配现有命名、结构、风格 |
| ❌ 实现完不验证 | ✓ 对照验收标准逐条验证 |

## 协作

| 场景       | 调用               |
|----------|------------------|
| 无文档/需要设计 | → `designing` 技能 |
| 需要代码审查   | → `reviewing` 技能 |

## 步骤 6：单元测试

### 测试策略

| 技术栈 | 要求 | 操作 |
|--------|------|------|
| **后端 Java** | **必须** | 自动为核心逻辑编写单元测试 |
| **前端 Vue/TS** | **可选** | 询问用户是否需要编写单元测试 |

### 后端 Java 单元测试（必须）

验证完成后，自动执行：

1. 为新增/修改的核心业务逻辑编写单元测试
2. 测试覆盖：正常流程、边界条件、异常情况
3. 运行测试确保全部通过

```
单元测试清单：
- [ ] 核心业务逻辑已覆盖
- [ ] 边界条件已测试
- [ ] 异常情况已处理
- [ ] 变异测试（重要，验证单测有效性）
- [ ] 所有测试通过
```

### 前端单元测试（可选）

询问用户："是否需要为前端代码编写单元测试？"

- **用户选择是**：编写组件/函数的单元测试
- **用户选择否**：跳过此步骤

## 步骤 7：代码优化（可选）

### 触发条件

询问用户："是否需要优化简化代码？"

- **用户选择是**：执行代码优化
- **用户选择否**：跳过此步骤

### 优化目标

使用 `code-simplifier` 代理进行代码优化：

| 优化项 | 说明 |
|--------|------|
| **消除重复** | 提取公共逻辑，减少代码冗余 |
| **统一风格** | 确保代码风格一致 |
| **提高可读性** | 改善命名、简化复杂逻辑 |
| **保持功能** | 优化不改变任何功能行为 |

### 优化流程

1. 调用 `code-simplifier` 代理分析并优化代码
2. **自动运行单元测试**验证功能未受影响
3. 如果测试失败：回滚优化，修复后重试
4. 循环直至所有测试通过

```
代码优化清单：
- [ ] 代码已优化简化
- [ ] 单元测试全部通过
- [ ] 功能行为未改变
```

## 不确定性处理

当对实现方法或 API 用法不确定时：

- 声明："我对 [具体方面] 不确定。让我验证一下。"
- 继续之前搜索官方文档或现有代码库
- 如果研究后仍不确定：问用户而不是猜测
