# 审查清单

审查 detail文档内容的准确性和完整性。

**注意**：这是文档审查，不是代码审查。专注于 detail 描述是否正确，而非代码实现细节。

## 使用方法

1. 复制下方清单到回复中
2. **逐条**对照代码和需求验证
3. 每条检查后**立即记录**结论（✓ 通过 / ✗ 问题 + 原因）
4. 禁止批量标记，必须逐条分析

## 审查清单模板

```markdown
## 审查记录

### 一、文档结构检查

- [ ] 1.1 包含"现状分析"章节
- [ ] 1.2 包含"整体思路"章节
- [ ] 1.3 任务项完整
- [ ] 1.4 方法签名完整（返回类型+方法名+参数）
- [ ] 1.5 未包含实现代码（禁止出现代码块）

### 二、现状分析验证

**要求：对照实际代码逐条验证描述是否属实、合理**

- [ ] 2.1 提及的模块/代码文件是否正确存在
- [ ] 2.2 描述的依赖关系/调用链是否与代码一致
- [ ] 2.3 是否遗漏了重要的现有实现

### 三、可行性质疑

**要求：基于代码现状，质疑整体思路是否可行**

- [ ] 3.1 整体思路是否与现有架构风格一致
- [ ] 3.2 整体思路是否是最佳方案
- [ ] 3.3 是否忽略了现有可复用的工具类/组件
- [ ] 3.4 是否有明显更简单的实现路径
- [ ] 3.5 是否存在技术债、workaround

### 四、任务覆盖检查

**要求：对照需求源（prd/trd/roadmap）验证任务覆盖度是否100%**

- [ ] 4.1 任务项是否100%覆盖了prd需求
- [ ] 4.2 任务之间的依赖顺序是否合理
- [ ] 4.3 是否遗漏了隐含任务（数据库变更、配置变更、权限配置等）

### 五、文件变更描述验证

**要求：逐项验证描述的文件/方法是否正确**

- [ ] 5.1 变更是否符合整体思路和任务思路
- [ ] 5.2 是否遗漏了需要变更的文件或方法(忽略实体的get/set方法)
- [ ] 5.3 是否有多余的文件变更，与当前任务无关

## 输出格式

````markdown
## 审查结果

| 编号 | 检查项 | 状态 | 问题描述 | 建议修改 |
|------|--------|------|---------|---------|
| 2.1 | 类名验证 | ✗ | UserManager 不存在，实际类名是 UserService | 修正类名 |
| 5.2 | 遗漏文件 | ✗ | 未包含 XxxController 的修改 | 新增该文件的变更说明 |
````
```

## 审查心态

- **验证描述，不审代码**：关注 detail 说的对不对
- **用代码验证文档**：每条描述都要找到对应代码确认
- **不确定时声明**："我无法确认 X，因为..."
