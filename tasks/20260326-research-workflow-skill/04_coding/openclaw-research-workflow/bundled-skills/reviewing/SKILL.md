# 审查

宁可误报也不能漏报。代码必须与需求/技术文档对齐。

## 工作流程

复制此清单：

```
审查进度：
- [ ] 步骤 1：确定范围（用户指定，未指定必须询问）
- [ ] 步骤 2：查找文档（需求文档、技术文档）
- [ ] 步骤 3：浏览代码结构（了解现状，不深入细节）
- [ ] 步骤 4：需求审查（有需求文档时必做）
- [ ] 步骤 5：架构审查（有技术文档时必做）
- [ ] 步骤 6：阅读最佳实践（如果有）
- [ ] 步骤 7：代码审查（深入代码细节，逐项检查）
- [ ] 步骤 8：输出报告
```

## 步骤 1：确定范围

用户指定的文件。**如果未指定，必须在继续之前询问用户审查范围。绝不能从 git status 或其他来源假设或推断范围。**

## 步骤 2：查找文档

查找用户指定目录下的文档：

- **需求文档**：prd、roadmap 或其他描述"做什么"的文档
- **技术文档**：trd、detail 或其他描述"怎么做"的文档

## 步骤 3：浏览代码结构

**目的**：了解代码现状，为需求/架构审查提供上下文。**不深入细节**。

浏览清单：

- [ ] 项目结构（模块划分、依赖关系）
- [ ] 关键文件和方法签名
- [ ] 与审查范围关联的业务代码

## 步骤 4：需求审查（有需求文档时必做）

基于步骤 3 了解的代码现状，审查需求文档。

见 [CHECKLISTS.md](CHECKLISTS.md) 需求审查清单。

## 步骤 5：架构审查（有技术文档时必做）

基于步骤 3 了解的代码现状，审查技术文档。

见 [CHECKLISTS.md](CHECKLISTS.md) 架构审查清单。

## 步骤 6：阅读最佳实践文档

**代码审查前必须阅读对应技术栈的最佳实践文档**

如何自动检测技术栈，见：[STACK-CHECK-LIST.md](../.best-practices/STACK-CHECK-LIST.md)

## 步骤 7：代码审查

**深入代码细节**，逐项检查清单。

见 [CHECKLISTS.md](CHECKLISTS.md) 代码审查清单。

## 步骤 8：输出报告

见 [REPORT-TEMPLATES.md](REPORT-TEMPLATES.md) 获取输出格式。

**结论**：通过 / 不通过，附清晰理由。

## 核心原则

| 原则    | 描述                  |
|-------|---------------------|
| 文档对齐  | 与需求/技术文档不符 = 问题 |
| 双向修复  | 需求错 → 改文档；代码错 → 改代码 |
| 从严不从宽 | 不确定时，报告它            |

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| ❌ 未指定范围就从 git status 推断 | ✓ 必须询问用户确认审查范围 |
| ❌ 批量标记"通过"不逐条检查 | ✓ 逐条执行清单，立即记录结论 |
| ❌ 发现问题不报告怕误报 | ✓ 从严不从宽，宁可误报不漏报 |
| ❌ 只看代码不对照文档 | ✓ 每个实现必须与需求/技术文档对照 |
| ❌ 有需求文档却跳过需求审查 | ✓ 有需求文档必做步骤 4 |
| ❌ 有技术文档却跳过架构审查 | ✓ 有技术文档必做步骤 5 |
| ❌ 不确定就跳过 | ✓ 标记置信度后报告 |

## 不确定性处理

当不确定是否是问题时：

- 默认：报告它（从严不从宽）
- 标记置信度："潜在问题（需验证）：[描述]"
- 如果无法判断影响："我没有足够的上下文来评估严重程度。请澄清 [具体问题]。"
