# 审查清单

## 需求审查（有需求文档时必做）

阅读需求文档（prd、roadmap 等），逐条检查：

```
需求审查：
- [ ] 这个功能有必要吗？
- [ ] 需求是否自洽？
- [ ] 边界情况覆盖了吗？
- [ ] 各需求文档之间是否一致？
```

## 架构审查（有技术文档时必做）

阅读技术文档（trd、detail 等），逐条检查：

```
架构审查：
- [ ] 有更简单的方法吗？
- [ ] 需求变更时要改多少？
- [ ] 模块职责清晰吗？
- [ ] 技术文档与需求文档是否一致？
```

## 代码审查

### 🚫 阻塞级别（必须解决才能通过）

```
阻塞检查：
- [ ] 需求与代码实现不符（如果有文档） ← 重点关注
- [ ] 缺失功能（需求中定义但未实现）
- [ ] 正常路径不工作（端到端流程走不通）
- [ ] 返回值不符合接口定义
- [ ] 安全漏洞（注入、XSS、泄露的密钥、硬编码的密钥）
- [ ] 数据丢失风险（删除未处理关联数据、不可恢复）
- [ ] 类型错误（滥用 any 泛型、类型断言绕过）
```

### ⚠️ 必报级别（必须报告）

```
必报检查：
- [ ] 入口点不可发现（功能存在但用户找不到）
- [ ] 缺失错误处理（未捕获的异常、崩溃而非优雅降级）
- [ ] 未处理的边界（null、空、满、极端值）
- [ ] 数据校验缺失（必填字段未校验、默认值不合理）
- [ ] 废弃/过时的 API
- [ ] 代码重复（相同逻辑出现 2 次以上）
- [ ] 死代码（未使用的代码、过时的注释）
- [ ] 过度设计（不必要的抽象、过早优化）
- [ ] 设计不足（缺少必要的抽象、硬编码、魔法数字）
- [ ] 竞态条件（并发冲突、部分更新不安全）
- [ ] 资源泄漏（内存/连接未释放）
- [ ] 不符合技术栈最佳实践（如果存在最佳实践文档）
- [ ] 函数过大（巨型怪物而非小而专注）
- [ ] 深度嵌套（分支流程不清晰）
- [ ] 职责越界（模块边界不清、紧耦合）
- [ ] 隐藏副作用（状态变更不透明）
- [ ] 事务不完整（部分状态而非原子操作）
- [ ] 非幂等操作（重试会出问题）
- [ ] 无法回滚（变更卡住无法撤退）
```

### 💡 建议级别（可选报告）

```
建议检查：
- [ ] 命名不清晰（晦涩难懂而非自解释）
- [ ] 注释问题（噪音而非澄清）
- [ ] 性能优化机会
- [ ] 日志/可观测性不足
- [ ] 过度防御（多余的检查、重复的验证）
- [ ] 可测试性差（代码缠绕难以测试）
- [ ] 风格不一致（不符合项目规范）
```
