# 示例 3: 问题修复

## 场景说明

修复用户登录会话在 24 小时内无故过期的问题，包括问题定位、修复实施、监控添加等。

## 使用方式

```bash
/sessions_spawn clawdevflow

# 任务：修复登录会话过期问题
# 场景类型：问题修复
# 问题描述：
#   用户反馈登录会话在 24 小时内无故过期，需要：
#   - 定位问题原因（JWT 配置/Redis 过期/代码逻辑）
#   - 修复会话管理逻辑
#   - 添加会话监控和日志
#   - 防止类似问题再次发生
# 现象：
#   - 用户登录后 2-24 小时内随机被登出
#   - 不是所有用户都受影响（约 30%）
#   - 重新登录后可正常使用
#   - 无错误日志
# 现有系统：
#   - JWT 过期时间：2 小时
#   - Refresh Token 过期时间：7 天
#   - Redis 存储黑名单（已登出的 token）
#   - Redis 过期时间：7 天
# 约束条件：
#   - 不影响现有功能
#   - 最小化代码变更
#   - 添加日志便于排查
#   - 不能降低安全性
# 验收标准：
#   - 会话在 7 天内保持稳定（无主动登出）
#   - 过期前有明确提示（提前 1 小时）
#   - 日志记录会话生命周期（创建/刷新/过期/登出）
#   - 添加会话监控指标
#   - 修复后无回归问题
```

## 预期输出

### 阶段 1: designing

**输出文件**:
- `01_designing/PRD.md` - 问题修复需求文档
- `01_designing/TRD.md` - 问题修复技术文档

**PRD.md 包含**:
- 问题描述和现象
- 影响范围（30% 用户，2-24 小时）
- 修复目标（7 天稳定）
- 验收标准
- 风险评估

**TRD.md 包含**:
- 问题分析（可能的原因）
- 排查方案（日志/监控/复现）
- 修复方案（代码变更）
- 监控方案（指标/告警）
- 回滚方案

### 阶段 2: roadmapping

**输出文件**:
- `02_roadmapping/ROADMAP.md` - 修复计划

**包含**:
- 排查阶段（日志分析、复现问题）
- 修复阶段（代码变更、测试）
- 验证阶段（灰度发布、监控）
- 时间估算（排查 4h、修复 2h、验证 4h）

### 阶段 3: detailing

**输出文件**:
- `03_detailing/DETAIL.md` - 修复详细设计

**包含**:
- 问题定位步骤
- 代码变更位置
- 日志添加位置
- 监控指标定义
- 测试用例

## 完整流程

```
问题描述
  ↓
designing → 问题分析 → PRD.md(修复) + TRD.md(修复)
  ↓
roadmapping → 修复计划 → ROADMAP.md
  ↓
detailing → 修复详细设计 → DETAIL.md
  ↓
coding → 修复代码 + 测试 + 日志 + 监控
  ↓
reviewing → 修复验收报告
  ↓
完整研发产出（修复）
```

## 问题排查流程

### 步骤 1: 收集信息

- 用户反馈时间分布
- 受影响用户特征
- 服务器日志分析
- 监控系统数据

### 步骤 2: 复现问题

- 尝试在测试环境复现
- 模拟高并发场景
- 检查边界条件

### 步骤 3: 定位原因

可能的原因：
1. JWT 配置错误（过期时间计算）
2. Redis 过期时间设置错误
3. 时区问题导致时间计算错误
4. 并发问题导致 token 被误删
5. 服务器时间不同步

### 步骤 4: 实施修复

- 最小化代码变更
- 添加充分日志
- 编写测试用例

### 步骤 5: 验证修复

- 单元测试通过
- 集成测试通过
- 灰度发布验证
- 监控指标正常

## 关键检查点

1. **问题分析**: 确认理解问题现象和影响范围
2. **修复方案**: 确认修复方案治标治本
3. **回归测试**: 确认修复不引入新问题
4. **监控验证**: 确认监控指标能发现问题
5. **验收通过**: 确认问题彻底解决

## 与全新功能开发的区别

| 方面 | 全新功能 | 问题修复 |
|------|----------|----------|
| **designing** | 完整设计 | 问题分析 + 修复方案 |
| **优先级** | 按计划 | 高优先级（P0/P1） |
| **测试** | 完整测试 | 修复测试 + 回归测试 |
| **部署** | 正常发布 | 紧急修复/热修复 |
| **监控** | 常规监控 | 专项监控 + 告警 |

## 常见问题

**Q: 如何快速定位问题？**

A: 添加详细日志，使用监控指标，在测试环境复现。

**Q: 修复引入新问题怎么办？**

A: 完整的回归测试，灰度发布，快速回滚机制。

**Q: 如何防止类似问题再次发生？**

A: 复盘报告，更新检查清单，添加自动化测试。

```markdown
## 改进措施
{防止再次发生的措施}

{学到的经验}
```
