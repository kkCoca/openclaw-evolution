# 执行方案生成与审阅

根据用户输入，生成或审查 detail 文档，并执行自审阅。

## 步骤 1：检测模式

| 模式 | 触发条件 | 工作流 |
|------|---------|--------|
| **新建** | 无 detail.md 或用户要求重新生成 | 使用 [新建工作流](#新建工作流) |
| **审查** | 有 detail.md，用户要求审查 | 使用 [审查工作流](#审查工作流) |

---

## 新建工作流

复制此清单：

```
新建方案进度：
- [ ] 步骤 1：理解输入（prd/trd/roadmap/用户描述）
- [ ] 步骤 2：确认生成范围（全栈/前端/后端/部分）
- [ ] 步骤 3：阅读相关代码，理解现状
- [ ] 步骤 4：阅读最佳实践（如果有）
- [ ] 步骤 5：按模板生成执行方案
- [ ] 步骤 6：写入 detail 文档
- [ ] 步骤 7：执行自审阅（10 项检查清单）
- [ ] 步骤 8：修正（如需要）
```

### 步骤 2：读取输入

prd/trd/roadmap/用户描述

### 步骤 3：确认生成范围

使用交互选择工具询问：
- 全栈（所有任务）
- 仅前端任务
- 仅后端任务
- 指定任务

### 步骤 4：阅读相关代码

1. 定位涉及的模块/文件
2. 理解代码现状和依赖关系
3. 确定需要新增/修改/删除的文件和方法


### 步骤 5：阅读最佳实践（如果有）

**实施前必须阅读对应技术栈的最佳实践文档**

如何自动检测技术栈，见：[STACK-CHECK-LIST.md](../.best-practices/STACK-CHECK-LIST.md)

### 步骤 6：生成执行方案

按模板生成：**阅读 [TEMPLATES.md](./TEMPLATES.md)**

关键约束：
- 按文件组织，标注 `[新增]`/`[修改]`/`[删除]`
- 方法签名必须完整
- **禁止写实现代码**

### 步骤 7：自审阅（关键！）

生成 DETAIL.md 后，**必须**执行自审阅，逐项检查以下清单：

#### 审阅检查清单（10 项）

**Hard Gates（Critical 项，一票否决）**：
- HG1 ✅ **Freshness 对齐** - DETAIL.md 是否包含 alignedTo + requirementsHash 字段？哈希是否与 REQUIREMENTS.md 一致？
- HG2 ✅ **需求可追溯** - DETAIL 是否包含需求追溯矩阵？每个需求是否有可定位的映射（章节 + 行号）？覆盖率 100%？
- HG3 ✅ **验收可测试** - 验收标准是否包含 Given/When/Then 格式？无模糊术语？
- D0 ✅ **章节完整性** - 是否包含所有必需章节（设计概述/需求追溯/架构设计/模块设计/接口定义/数据结构/验收标准）？
- D2 ✅ **技术一致性** - 是否与 TRD.md 的技术选型一致？无冲突？

**Normal 项（允许条件通过）**：
- D3 ⚠️ **计划对齐** - DETAIL 任务是否与 ROADMAP.md 一致？无遗漏/无新增？
- D4 ⚠️ **接口定义完整性** - 所有接口是否有完整定义（参数/返回值/错误码）？
- D5 ⚠️ **数据结构设计** - 数据结构是否完整（字段/类型/约束）？
- D6 ⚠️ **异常处理** - 异常处理是否完整（正常/失败/边界）？
- D7 ⚠️ **向后兼容** - 增量需求时是否无破坏性变更？

#### 审阅评分

**评分规则**：
1. **检查 Hard Gates（HG1-HG3 + D0 + D2）**：
   - 任一 Critical 项失败 → **驳回**（重新生成）
   - 全部 Critical 项通过 → 继续检查 Normal 项

2. **检查 Normal 项（D3-D7）**：
   - 全部通过 → **通过**（10/10）
   - 1-2 项失败 → **条件通过**（8-9/10，修正后输出）
   - 3 项及以上失败 → **驳回**（<8/10，重新生成）

**审阅决策**：
- **10/10** - 所有检查项符合标准 → 直接通过
- **8-9/10** - 1-2 项 Normal 问题 → 修正后通过
- **<8/10 或 Critical 失败** - 3 项及以上问题或 Critical 项失败 → 重新生成

#### 输入规范（必须读取 5 个文件）

| 文件 | 用途 | 说明 |
|------|------|------|
| REQUIREMENTS.md | 最新需求 source of truth | 验证需求对齐 |
| PRD.md | 产品需求 | 验证功能完整性 |
| TRD.md | 技术设计 | 验证技术一致性 |
| ROADMAP.md | 开发计划 | 验证计划对齐 |
| DETAIL.md | 被审阅对象 | 详细设计文档 |

#### 输出规范（必须生成 3 项）

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 审阅结论 | pass/conditional/reject | 明确决策 |
| 失败项列表 | 章节/行号 | 明确到具体位置 |
| 修复建议 | 示例 | 指出如何修复 |

### 步骤 8：修正（如需要）

如果自审阅评分 <10 分：
1. 记录问题到 DETAIL-SELF-REVIEW.md（仅 Critical 失败时生成）
2. 修正 DETAIL.md
3. 重新审阅（最多 3 次）

**修正示例**：
```
问题：HG2 需求可追溯 - REQ-013 缺少 DETAIL 章节映射
修正：在需求追溯矩阵中添加行：| REQ-013 | L780-900 | 18.1-18.6 | 15.1-15.4 | 505-600 | ✅ |

问题：HG3 验收可测试 - 缺少 Given/When/Then 格式
修正：将验收标准改写为：
  - Given 用户已登录
  - When 用户访问受保护资源
  - Then 返回资源内容
```

### 步骤 9：写入文件

将最终的详细设计写入 `${input.outputDir || '03_detailing'}/DETAIL.md`。

**DETAIL-SELF-REVIEW.md 生成逻辑**（v3.1.8 优化）：
- ✅ **Critical 项（HG1-HG3, D0, D2）全部通过** → 不生成 SELF-REVIEW.md
- ✅ **仅 Normal 项失败** → 不生成 SELF-REVIEW.md（修正后直接输出）
- ⚠️ **Critical 项任一失败** → 生成 DETAIL-SELF-REVIEW.md（记录失败原因用于调试）

**优化理由**：符合"不生成额外文件"原则，减少文件冗余

## 审查工作流

复制此清单：

```
审查方案进度：
- [ ] 步骤 1：理解输入（prd/trd/roadmap/用户描述）
- [ ] 步骤 2：阅读 CHECKLIST.md
- [ ] 步骤 3：复制 CHECKLIST 到回复
- [ ] 步骤 4：阅读最佳实践（如果有）
- [ ] 步骤 5：逐条检查并立即记录结论
- [ ] 步骤 6：输出问题汇总表
- [ ] 步骤 7：用户确认后逐一修复
```

### 步骤 2：理解输入

prd/trd/roadmap/用户描述

### 步骤 3：执行审查

**必须阅读：[CHECKLIST.md](./CHECKLIST.md)**

审查原则：
- **逐条执行**：不可跳过任何检查项
- **立即记录**：每条检查后立即写下结论
- **代码验证**：每条描述必须与实际代码对照
- **自行验证**：需确认 xx 是否 xx，深入分析自行确认而不是抛给用户
- **不确定时声明**："深入分析后我仍然无法确认 xxx，因为..."

**实施前必须阅读对应技术栈的最佳实践文档**

如何自动检测技术栈，见：[STACK-CHECK-LIST.md](../.best-practices/STACK-CHECK-LIST.md)

### 步骤 4：输出问题汇总

**建议修改**：给出明确的最佳实践方案，而不是抛出多个选项

```markdown
## 审查结果

| 编号 | 检查项 | 状态 | 问题描述 | 建议修改 |
|------|--------|------|---------|---------|
```

### 步骤 5：修复问题

用户确认后，逐一修复问题列表中的项。

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| ❌ 不读代码就猜测文件结构 | ✓ 先 Grep/Glob 定位，再 Read 验证 |
| ❌ 在方案中写实现代码 | ✓ 只写方法签名和说明 |
| ❌ 审查时批量标记"通过" | ✓ 逐条检查，立即记录 |
| ❌ 跳过"不确定"的检查项 | ✓ 明确声明不确定及原因 |
| ❌ 遗漏配置/数据库变更 | ✓ 检查附录是否完整 |
| ❌ 自审阅不执行 Hard Gates 检查 | ✓ 必须执行 HG1-HG3，一票否决 |
| ❌ 需求追溯矩阵缺少行号 | ✓ 必须包含章节和行号，可定位 |
| ❌ 验收标准无 Given/When/Then | ✓ 必须使用 Given/When/Then 格式 |
