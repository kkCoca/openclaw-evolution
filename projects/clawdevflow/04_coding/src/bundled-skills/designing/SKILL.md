# 设计

## AI 工具配置

**重要**: 本 skill 由流程引擎调用，AI 工具选择根据 config.yaml 配置。

**配置文件**: `04_coding/src/config.yaml`

**配置说明**:
```yaml
global:
  # 默认 AI 工具
  defaultAITool: opencode  # 可选：opencode / claude-code / custom

stages:
  designing:
    # 阶段专用 AI 工具（可覆盖全局配置）
    aiTool: opencode
```

**输出要求**:
- PRD.md 和 TRD.md 中描述 AI 工具时，使用"根据 config.yaml 配置选择 AI 工具"
- 不要硬编码具体 AI 工具名称（如"使用 OpenCode"）
- 正确示例："根据 config.yaml 配置选择 AI 工具（默认 opencode）"
- 错误示例："使用 OpenCode 执行 designing skill"

## 工作流程

复制此清单：

```
设计进度：
- [ ] 步骤 1：启动分析（识别输入类型 → 确定输出要求 → 检测模式）
- [ ] 步骤 2：与用户确认模式和输出格式
- [ ] 步骤 3：询问需求（仅在需要时，一次一个问题）
- [ ] 步骤 4：按要求输出
- [ ] 步骤 5：一致性检查（多文档时）
```

## 步骤 1：启动分析

**识别输入类型**：
- 用户输入是简短描述 → 需要询问完善
- 用户输入已经是完整 prd → 跳过询问
- 用户输入是现有文档 → 审查/增量模式

**确定输出要求**（直接问用户或从上下文推断）：
- 一段话（快速总结）
- 单个 prd（小需求默认）
- 单个 trd
- prd + trd（大需求默认）
- 自定义格式（N 个 md 文档）

**小需求判断**（满足多数即为小需求，默认只输出 prd）：
- 功能点少（1-2 个简单功能）
- 不涉及架构变更
- 无需新技术栈引入
- 实现方案显而易见
- 不涉及跨模块协作

**检测模式**：

| 模式     | 触发条件           | 流程                                              |
|--------|----------------|-------------------------------------------------|
| **新建** | 无文档，用户描述新项目    | 完整询问 → 按用户要求输出                                  |
| **直出** | 用户输入已是完整需求/prd | 跳过询问 → 按用户要求输出                                  |
| **审查** | 有文档，用户要求审查     | 阅读 → 诊断 → 修复列表。见 [DIAGNOSIS.md](./DIAGNOSIS.md) |
| **增量** | 有代码，用户添加功能     | 分析现有 → 针对性询问 → 按用户要求输出                          |

## 步骤 2：确认

继续前与用户确认模式和输出格式。

## 步骤 3：询问需求

**一次一问，先建议后提问，提供选项让用户快速选择。**

```
[主题]
建议：[选项] - [理由]
比较：| 选项 | 优点 | 缺点 |
[问题]
```

**角色**：质疑必要性、挑战复杂性、探查边界、指出矛盾、纠正技术选择、填补空白。

**询问维度**：

- **功能**：正常流程 → 失败处理 → 边界情况 → 状态转换
- **数据**：字段 → 类型约束 → 实体关系 → 生命周期

**终止条件**：用户说"ok" 或者 正常 + 失败 + 边界情况都覆盖了

## 步骤 4：按要求输出

如果需要模板化输出 prd/trd 文档，见 [TEMPLATES.md](TEMPLATES.md)

**技术选型要求**：

1. 推荐前搜索当前最佳实践
2. 每个选择必须有比较表
3. 禁止：已弃用 / 不兼容 / 过时的方案

## 步骤 5：一致性检查

**生成前自检**（全部必须为真）：

- 覆盖率 100%？（相对于选定的输出类型）
- 所有失败路径都问了？（直出模式例外）
- 所有字段都确认了？（直出模式例外）
- 没有未确认的假设？
- 没有模糊描述？

**生成后检查**（多文档输出时）：

- 各文档术语一致
- 需求文档 ↔ 技术文档
- 类型引用 ↔ 类型定义

发现问题 → 主动询问，不要等用户发现

## 常见陷阱

| 陷阱                   | 正确做法                             |
|----------------------|----------------------------------|
| ❌ 假设用户需求而不确认         | ✓ 每个关键决策都要确认                     |
| ❌ 一次问多个问题            | ✓ 一次一问，降低用户负担                    |
| ❌ 跳过失败路径的询问          | ✓ 正常流程、失败处理、边界都要覆盖               |
| ❌ 未确认就直接写入文档         | ✓ 先确认再输出                         |
| ❌ 使用模糊术语（"适当的"、"一些"） | ✓ 所有描述必须具体可测试                    |
| ❌ prd/trd 术语不一致      | ✓ 输出前执行一致性检查                     |
| ❌ prd/trd 职责混乱       | ✓ prd只应该包含需求+UI设计，trd只包含技术+数据库设计 |
| ❌ trd代码细节过多          | ✓ 只包含重要的枚举、接口定义、方法签名             |

## 不确定性处理

当对技术决策或领域知识不确定时：

- 声明："我对 [具体方面] 不确定。我的理解是 [X]，但需要验证。"
- 对于技术选择：推荐前搜索当前最佳实践
- 如果是领域特定的："我需要更多关于 [业务逻辑/领域规则] 的上下文才能正确设计。"

## 完成标准

| 条件       | 说明             |
|----------|----------------|
| 覆盖率 100% | 相对于选定输出类型的必需内容 |
| 无模糊术语    | 所有输出类型都适用      |
| 一致性通过    | 多文档输出时检查       |
| 自检通过     | 按模式适用的检查项      |
| 用户确认     | 最终确认           |

前 4 项不可协商
