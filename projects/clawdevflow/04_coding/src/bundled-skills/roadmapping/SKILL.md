# Roadmapping - 开发计划生成

**定位**：ClawDevFlow 流程引擎的第二阶段，**纯自动化执行 + 自审阅**，无用户交互。

**输入**：
- `01_designing/PRD.md` - 产品需求文档
- `01_designing/TRD.md` - 技术设计文档
- 项目源代码目录（增量需求时）

**输出**：
- `02_roadmapping/ROADMAP.md` - 开发计划文档
- `02_roadmapping/SELF-REVIEW.md` - 自审阅报告（可选）

---

## 执行流程

```
阅读 PRD.md → 阅读 TRD.md → 分析代码现状 → 生成开发计划 → 自审阅 → 修正 → 写入 ROADMAP.md
```

### 步骤 1：阅读设计文档

**必须阅读以下文件**：

| 文件 | 用途 |
|------|------|
| `01_designing/PRD.md` | 理解功能需求、业务边界、验收标准 |
| `01_designing/TRD.md` | 理解技术架构、模块划分、接口设计 |

**阅读重点**：
- PRD.md：功能列表、用户场景、约束条件
- TRD.md：技术栈、模块划分、数据模型、接口定义

---

### 步骤 2：分析代码现状（增量需求必须）

**仅增量需求执行此步骤**，全新功能跳过。

**自动化代码分析**：

```javascript
// 伪代码示例：扫描项目结构
const projectStructure = {
  src: {
    modules: ['auth', 'user', 'order'],
    controllers: ['auth.controller.js', 'user.controller.js'],
    services: ['auth.service.js', 'user.service.js'],
    models: ['user.model.js', 'order.model.js'],
    utils: ['logger.js', 'validator.js']
  },
  package.json: {
    dependencies: ['express', 'mongoose', 'jsonwebtoken'],
    devDependencies: ['jest', 'eslint']
  }
};
```

**分析内容**：
1. **目录结构** - 识别模块组织方式
2. **核心文件** - 识别控制器、服务、模型等
3. **技术栈** - 从 package.json/requirements.txt 识别依赖
4. **代码规模** - 统计文件数量、代码行数（粗略）

**目的**：了解改动基础，评估增量工作量。

---

### 步骤 3：生成开发计划

#### 任务拆分原则

| 维度 | 说明 |
|------|------|
| **按模块拆分** | 每个独立模块作为一个任务单元 |
| **按前后端拆分** | 前端任务和后端任务分开估算 |
| **按依赖关系拆分** | 有依赖关系的任务分先后顺序 |

#### 任务命名格式

固定格式，30 字以内：`【任务简称】(前端/后端) 任务简述`

示例：
- `【用户认证】(后端) 实现 JWT 登录接口`
- `【用户认证】(前端) 登录页面开发与对接`

#### 输出模板

```markdown
# 开发计划 - {项目名称}

## 需求概述

{基于 PRD.md 一句话描述需求目标}

## 代码现状（增量需求）

{简述现有代码结构、核心模块、技术栈}

## 开发计划

| 任务名称 | 详细描述 | 计划工作量 (人天) |
|----------|----------|------------------|
| 【任务 1】(后端)xxx | 1. 要点 1<br>2. 要点 2 | 1 |
| 【任务 1】(前端)xxx | 1. 要点 1<br>2. 要点 2 | 0.5 |
| ... | ... | ... |
| 【任务 1】(后端)联调测试 | 1. 后端接口联调<br>2. 功能测试 | 0.5 |
| 【任务 1】(前端)联调测试 | 1. 前端功能联调<br>2. 交互测试 | 0.5 |
| 【任务 1】演示 | 1. 功能演示准备<br>2. 演示环境验证 | 0.5 |
| **合计** | | **X** |

## 涉及模块

- 模块 1: 说明
- 模块 2: 说明

## 风险评估

| 风险项 | 可能性 | 影响 | 应对措施 |
|--------|--------|------|---------|
| {风险 1} | 高/中/低 | 高/中/低 | {应对措施} |
```

#### 详细描述要求

- ✅ 只描述"做什么"，不涉及"怎么做"
- ✅ 按条理用无序列表组织
- ✅ 简洁清晰，不超过 300 字
- ✅ 包含验收要点（可选）

---

### 步骤 4：自审阅（关键！）

生成 ROADMAP.md 后，**必须**执行自审阅，确保质量。

#### 审阅检查清单

**必须逐项检查**以下项目：

| 检查项 | 检查内容 | 标准 |
|--------|---------|------|
| ✅ 任务拆分 | 任务是否按模块和前后端拆分？ | 每个任务职责单一 |
| ✅ 工作量评估 | 单个任务是否 ≤ 2 人天？ | 超过 2 人天的任务已拆分 |
| ✅ 收尾项 | 是否包含联调测试和演示项？ | 每个任务都有联调测试（0.5 人天）+ 演示（0.5 人天） |
| ✅ 任务命名 | 是否使用固定格式？ | `【任务简称】(前端/后端) 任务简述` |
| ✅ 描述规范 | 是否只描述"做什么"？ | 不涉及"怎么做"的技术实现 |
| ✅ 需求覆盖 | 是否覆盖 PRD.md 中的所有功能？ | 无遗漏 |
| ✅ 技术对齐 | 是否与 TRD.md 的技术选型一致？ | 无冲突 |
| ✅ 代码现状（增量） | 是否分析了现有代码？ | 增量需求必须包含代码现状章节 |
| ✅ 风险评估 | 是否识别了主要风险？ | 至少 3 项风险 |
| ✅ 不确定性标注 | 不确定的任务是否标注？ | 标注原因和范围估算 |

#### 审阅评分

根据检查清单计算得分：

- **10/10 通过** - 所有检查项符合标准
- **8-9/10 条件通过** - 1-2 项小问题，不影响整体
- **<8/10 驳回** - 3 项及以上问题，需要修正

#### 审阅决策

| 评分 | 决策 | 后续动作 |
|------|------|---------|
| 10/10 | ✅ 通过 | 直接写入 ROADMAP.md |
| 8-9/10 | ⚠️ 条件通过 | 修正问题后写入 ROADMAP.md |
| <8/10 | ❌ 驳回 | 重新生成开发计划 |

---

### 步骤 5：修正（如需要）

如果自审阅评分 <10 分：

1. **记录问题** - 将问题写入 `02_roadmapping/SELF-REVIEW.md`
2. **修正计划** - 根据审阅意见修正 ROADMAP.md
3. **再次审阅** - 重新执行步骤 4（最多 3 次）

**修正示例**：

```
问题：任务【用户管理】(后端) 工作量为 3 人天，超过 2 人天上限
修正：拆分为【用户管理】(后端) 用户 CRUD 接口（1.5 人天）+ 【用户管理】(后端) 权限控制接口（1 人天）
```

---

### 步骤 6：写入文件

将最终的开发计划写入 `02_roadmapping/ROADMAP.md`。

**文件格式**：Markdown

**编码**：UTF-8

**可选**：如果执行过修正，同时写入 `02_roadmapping/SELF-REVIEW.md` 记录审阅过程。

## 工作量评估原则

### 基准原则

| 任务类型 | 工作量范围 | 说明 |
|----------|-----------|------|
| 简单任务 | 0.5 ~ 1 人天 | 单一功能点，无复杂逻辑 |
| 中等任务 | 1 ~ 1.5 人天 | 涉及多个模块，有一定复杂度 |
| 复杂任务 | 1.5 ~ 2 人天 | 跨模块、涉及核心逻辑 |

### 拆分规则

- ✅ 单个任务不超过 2 人天
- ✅ 超过 2 人天的任务必须拆分为子任务
- ✅ 尽量控制在 1 人天以内

### 评估依据

| 因素 | 增加工作量 | 减少工作量 |
|------|-----------|-----------|
| **代码现状** | 遗留代码复杂、无测试 | 代码清晰、有完整测试 |
| **技术栈** | 新技术、不熟悉 | 熟悉的技术栈 |
| **依赖关系** | 外部依赖多、接口不稳定 | 内部依赖、接口稳定 |
| **需求清晰度** | 需求模糊、边界不清 | 需求清晰、验收标准明确 |

---

## 必须包含的收尾项

### 联调测试项

根据任务类型决定：

| 任务类型 | 联调测试项 | 工作量 |
|----------|-----------|--------|
| 全栈任务 | (后端) 联调测试 + (前端) 联调测试 | 各 0.5 人天 |
| 仅后端 | (后端) 开发自测 | 0.5 人天 |
| 仅前端 | (前端) 开发自测 | 0.5 人天 |

### 演示项

**必须添加**，固定工作量 **0.5 人天**。

```
| 【任务简称】演示 | 1. 功能演示准备<br>2. 演示环境验证 | 0.5 |
```

## 自动化要求

### 禁止事项

- ❌ **禁止用户交互** - 不询问、不确认、不评审
- ❌ **禁止调用其他 skill** - 独立完成，不依赖 designing 等技能
- ❌ **禁止主观猜测** - 基于文档和代码分析，不臆测

### 必须事项

- ✅ **自动阅读文档** - PRD.md + TRD.md 作为唯一输入源
- ✅ **自动分析代码** - 增量需求时自动扫描项目结构
- ✅ **自动输出文件** - 直接写入 ROADMAP.md，不等待确认
- ✅ **不确定性标注** - 对不确定的工作量标注说明
- ✅ **自动审阅** - 生成后必须执行 10 项检查清单

---

## 不确定性处理

当对工作量评估不确定时：

1. **声明不确定性**：
   ```
   > 注：我对 [具体任务] 的工作量不确定，因为 [原因]。
   ```

2. **给出范围估算**：
   ```
   预估 1~2 人天，取决于 [因素]
   ```

3. **标注风险**：
   在"风险评估"章节记录该不确定性。

## 与流程引擎的协作

### 审阅机制

Roadmapping 阶段完成后，流程引擎会：

1. 暂停执行，等待用户审阅
2. 用户审阅结论：通过/条件通过/驳回/需澄清/终止
3. 根据结论决定：进入下一阶段 / 重新执行当前阶段

### 与 Designing 阶段的衔接

- Designing 阶段输出：`PRD.md` + `TRD.md`
- Roadmapping 阶段输入：`PRD.md` + `TRD.md`
- **不重复设计** - 只基于设计文档生成计划，不修改设计

### 与 Detailing 阶段的衔接

- Roadmapping 阶段输出：`ROADMAP.md`
- Detailing 阶段输入：`ROADMAP.md` + `PRD.md` + `TRD.md`
- **任务粒度** - Roadmap 中的任务拆分应足够粗，Detailing 再细化

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| ❌ 不读文档就生成计划 | ✓ 必须先阅读 PRD.md + TRD.md |
| ❌ 增量需求不分析代码 | ✓ 增量需求必须扫描代码现状 |
| ❌ 单个任务超过 2 人天 | ✓ 拆分为子任务 |
| ❌ 遗漏联调测试和演示项 | ✓ 必须包含收尾项 |
| ❌ 详细描述写技术实现 | ✓ 只描述"做什么" |
| ❌ 任务命名不规范 | ✓ 使用固定格式 |
| ❌ 向用户询问信息 | ✓ 基于已有文档和代码分析 |
