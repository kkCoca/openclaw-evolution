# 开发计划生成

根据用户提供的需求，生成结构化的开发计划。

## 工作流程

复制此清单：

```
开发计划生成进度：
- [ ] 步骤 1：验证输入（需求描述或文档）
- [ ] 步骤 2：确认评估范围（全栈/前端/后端）
- [ ] 步骤 3：阅读代码现状（增量需求必须）
- [ ] 步骤 4：调用 designing skill 澄清需求
- [ ] 步骤 5：生成开发计划（按模板输出）
- [ ] 步骤 6：写入 roadmap.md
- [ ] 步骤 7：用户评审确认
```

## 步骤 1：验证输入

**必须**确认用户提供了以下之一：

| 输入类型 | 示例                         |
|------|----------------------------|
| 需求描述 | 文字描述的功能需求                  |
| 文档文件 | .md、.doc、.docx、.txt、.pdf 等 |

**如果未提供**，使用交互工具询问。

## 步骤 2：确认评估范围

使用交互工具询问用户：

```
评估范围：
1. 全栈（前端 + 后端）
2. 仅前端
3. 仅后端
```

根据范围决定后续任务的拆分方式。

## 步骤 3：阅读代码现状

**增量需求必须执行此步骤**，在调用 designing skill 之前：

1. 搜索相关代码
2. 阅读核心文件，理解现有实现
3. 总结代码现状（模块位置、核心类、关键方法）

**目的**：了解改动基础，为需求澄清和工作量评估提供依据。

## 步骤 4：澄清需求

使用 Skill 工具调用 `designing` 技能进行需求澄清。

**重要约束**：
- 只澄清业务功能（功能边界、异常处理、兼容性），**不讨论技术实现**
- 代码细节自己阅读，不询问用户

## 步骤 5：生成开发计划

### 任务命名格式

固定格式，最好在30字以内：`【任务简称】(前端/后端)任务简述`

示例：`【辅助审批支持发起节点】(前端)配置态-发起者节点配置支持`

### 输出模板

````markdown
# 开发计划 - {需求名称}

## 需求概述

{一句话描述需求目标}

## 开发计划

| 任务名称 | 详细描述 | 计划工作量(人天) |
|------------|----------|------------|
| 【任务1】(后端)xxx | 1. 要点1<br>2. 要点2 | 1 |
| 【任务1】(前端)xxx | 1. 要点1<br>2. 要点2 | 0.5 |
| ... | ... | ... |
| 【任务1】(后端)联调测试 | 1. 后端接口联调<br>2. 功能测试 | 1 |
| 【任务1】(前端)联调测试 | 1. 前端功能联调<br>2. 交互测试 | 1 |
| 【任务1】演示 | 1. 功能演示准备<br>2. 演示环境验证 | 0.5 |
| **合计** | | **X** |

## 涉及模块

- 模块1: 说明
- 模块2: 说明
````

### 详细描述要求

- 只描述"做什么"，不涉及"怎么做"
- 按条理用无序列表组织
- 简洁清晰，不超过 300 字

## 步骤 6：写入文件

将开发计划写入 `roadmap.md`。

## 步骤 7：用户评审

输出文档后，询问用户：是否需要调整？

根据用户反馈修正，直到用户确认通过。

## 注意事项

- 工作量仅估算开发阶段
- 增量需求必须先阅读代码现状

## 工作量评估原则

- 单个任务工作量范围：0.5 ~ 2 人天
- 尽量控制在 1 人天以内
- 超过 1 人天的项，如果可以拆分则细化为多个子任务

## 必须包含的收尾项

### 联调测试项

根据评估范围决定：

| 评估范围 | 联调测试项 | 工作量 |
|----------|-----------|--------|
| 全栈（前端+后端） | (后端)联调测试 + (前端)联调测试 | 各 0.5~1 人天 |
| 仅前端 | (前端)开发自测 | 0.5~1 人天 |
| 仅后端 | (后端)开发自测 | 0.5~1 人天 |

### 演示项

**必须添加**，固定工作量 **0.5 人天**。

```
| 【任务简称】演示 | 1. 功能演示准备<br>2. 演示环境验证 | 0.5 |
```

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| ❌ 不读代码就估算增量需求 | ✓ 增量需求必须先阅读代码现状 |
| ❌ 单个任务工作量超过 2 人天 | ✓ 拆分为多个子任务，控制在 0.5~2 人天 |
| ❌ 遗漏联调测试和演示项 | ✓ 必须包含收尾项 |
| ❌ 详细描述写"怎么做" | ✓ 只描述"做什么"，不涉及技术实现 |
| ❌ 向用户询问代码细节 | ✓ 代码细节自己阅读，只澄清业务功能 |
| ❌ 任务命名不规范 | ✓ 使用固定格式：【任务简称】(前端/后端)任务简述 |

## 不确定性处理

当对工作量评估不确定时：

- 声明："我对 [具体任务] 的工作量不确定，因为 [原因]。"
- 给出范围估算而非精确值："预估 1~2 人天，取决于 [因素]"
- 如果缺少信息："需要进一步了解 [具体方面] 才能准确评估。"
