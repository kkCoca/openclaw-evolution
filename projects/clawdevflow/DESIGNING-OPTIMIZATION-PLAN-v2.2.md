# DESIGNING 阶段优化方案 v2.2 - 签字回填 PRD + 不生成额外文件

**日期**: 2026-04-02 13:23  
**版本**: v2.2  
**优先级**: P0

---

## 一、核心优化

### 1.1 优化原则

**不生成额外文件** - 避免文件过多导致信息不一致

**签字回填** - 用户签字确认直接回填到 PRD.md 或 REQUIREMENTS.md

**内容提炼** - 生成确认内容供用户确认，但不生成独立文件

---

### 1.2 优化对比

| 方案 | 文件数量 | 信息一致性 | 版本管理 | 推荐度 |
|------|---------|-----------|---------|--------|
| v2.0（三个文档） | 3 个额外文件 | ❌ 低（易不一致） | ❌ 复杂 | ❌ 不推荐 |
| v2.1（合并文档） | 1 个额外文件 | ⚠️ 中（可能不一致） | ⚠️ 中等 | ⚠️ 可接受 |
| **v2.2（签字回填）** | **0 个额外文件** ✅ | **✅ 高（单一来源）** ✅ | **✅ 简单** ✅ | **✅ 强烈推荐** |

---

## 二、优化方案

### 2.1 流程优化

```
PRD.md + TRD.md 生成完成
    ↓
ReviewDesignAgent v3.1.0 审查（100/100）✅
    ↓
提炼确认内容（内存中，不生成文件）
    ↓
展示给用户确认
    ↓
用户确认通过？
    ↓
是 → 签字回填到 PRD.md → 进入 roadmapping 阶段
否 → 返回 designing 重新设计
```

## 三、多版本迭代支持

**PRD.md 版本历史章节**:

```markdown

| 版本 | 日期 | 变更说明 | 用户确认 | 状态 |

| v1 | 2026-04-02 | 初始版本 | ✅ 张三 | 已归档 |

```

### 3.2 Git 版本管理

**Git 提交记录**:
```bash
# v1 版本
git add 01_designing/PRD.md
git commit -m "design: PRD v1 用户确认签字 - 张三"
git tag prd-v1

# v2 版本
git add 01_designing/PRD.md
git commit -m "design: PRD v2 用户确认签字 - 张三（修改后端技术选型）"
git tag prd-v2

# v3 版本（当前）
git add 01_designing/PRD.md
git commit -m "design: PRD v3 用户确认签字 - 待签字（新增 REQ-008）"
```

**版本对比**:
```bash
# 对比 v2 和 v3

```

## 七、总结

### 7.1 核心优化

1. **不生成额外文件** - 避免文件过多导致信息不一致
2. **签字回填 PRD.md** - 签字直接关联到需求文档
3. **确认内容内存提炼** - 供用户快速理解，不生成文件
4. **Git 版本管理** - 通过 Git 管理 PRD.md 版本

### 7.2 方案对比

| 特性 | v2.0（三个文档） | v2.1（合并文档） | v2.2（签字回填） |
|------|----------------|-----------------|-----------------|
| 文件数量 | 3 个额外文件 | 1 个额外文件 | **0 个额外文件** ✅ |
| 信息一致性 | ❌ 低 | ⚠️ 中 | **✅ 高** ✅ |
| 版本管理 | ❌ 复杂 | ⚠️ 中等 | **✅ 简单** ✅ |
| 签字关联 | ❌ 独立文件 | ⚠️ 独立文件 | **✅ PRD.md** ✅ |
| 推荐度 | ❌ 不推荐 | ⚠️ 可接受 | **✅ 强烈推荐** ✅ |

### 7.3 优势

- ✅ **文件极少** - 只有 PRD.md + TRD.md，无额外文件
- ✅ **信息单一来源** - 签字直接在 PRD.md 上，不易不一致
- ✅ **版本管理简单** - Git 管理 PRD.md 即可
- ✅ **用户友好** - 确认内容简洁，快速理解
- ✅ **流程简洁** - 确认→签字→回填→进入下一阶段
