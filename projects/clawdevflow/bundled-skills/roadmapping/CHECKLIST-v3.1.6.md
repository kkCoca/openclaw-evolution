> **版本**：v3.1.6（2026-04-02）

## 检查清单（12 项）

### Critical 项（一票否决）

#### R0: Freshness 对齐 ✅
**检查内容**：ROADMAP.md 是否包含 alignedTo 和 requirementsHash 字段？

**标准**：
- ✅ ROADMAP.md 包含 `alignedTo: v3.1.6`
- ✅ ROADMAP.md 包含 `requirementsHash: sha256:{64 位哈希}`

**示例**：
```markdown
## 文档元数据

| 字段 | 值 |
|------|-----|
| **alignedTo** | v3.1.6 |
| **requirementsHash** | sha256:f0e44912d5778703c30ce7921ceb25a81a454672 |
```

#### R1: Traceability（需求引用） ✅
**检查内容**：ROADMAP.md 是否显式引用 REQUIREMENTS.md 中的所有需求 ID？

**标准**：
- ✅ 显式引用所有需求 ID（REQ-001 ~ REQ-012）
- ✅ 需求覆盖率 100%

**检查方法**：
1. 提取 REQUIREMENTS.md 中的所有 REQ-XXX
2. 检查 ROADMAP.md 中是否包含每个 REQ-XXX
3. 计算覆盖率

#### R2: MVP 可交付性 ✅
**检查内容**：是否包含 MVP/Phase 1/里程碑 1 段落？

**标准**：
- ✅ 存在 MVP/Phase 1/里程碑 1 段落
- ✅ 包含 scope（范围）描述
- ✅ 包含验收标准
- ✅ 包含工作量估算

**示例**：
```markdown
## MVP（Phase 1）

### 范围
- REQ-001 基础功能
- REQ-002 核心流程

### 验收标准
- Given 用户访问系统

### 工作量

- 前端：3 人天

```

#### R3: 依赖与风险 ✅
**检查内容**：ROADMAP.md 是否有 Dependencies/Risks 段落？

**标准**：
- ✅ 包含 Dependencies（依赖）段落
- ✅ 包含 Risks（风险）段落
- ✅ 至少 3 项风险
- ✅ 每项风险有应对措施

**示例**：
```markdown
## Dependencies

- Node.js 22+ 环境
- OpenClaw sessions_spawn API

## Risks

| 风险项 | 可能性 | 影响 | 应对措施 |
|--------|--------|------|---------|
| DDG 反爬虫 | 中 | 高 | 添加请求延迟、合理 User-Agent |
| HTML 结构变更 | 中 | 中 | 使用稳定选择器、定期验证 |
| 网络超时 | 高 | 低 | 设置超时时间、实现重试 |
```

### Non-Critical 项（允许条件通过）

#### R4: 范围膨胀风险 ⚠️
**检查内容**：检测"可能/可选/未来"等关键词，**但检查是否有缓解措施**。

**标准**：
- ✅ 有关键词 + 有缓解措施 → 通过（不误报）
- ⚠️ 有关键词 + 无缓解措施 → warning（标注范围膨胀风险）
- ✅ 无关键词 → 通过

**关键词列表**：
- 可能、或许、大概、也许
- 可选、optional
- 未来、将来、后续、maybe

**缓解措施关键词**：
- 缓解、应对、措施、方案
- MVP 不包含、Phase 2、后续版本
- 已标注、已规划

**示例**：
```markdown
<!-- ✅ 通过（有缓解措施） -->
可选功能：用户头像上传（后续版本实现）
缓解措施：MVP 版本不包含此功能，已标注为 Phase 2

<!-- ⚠️ warning（无缓解措施） -->
可能需要在未来考虑性能优化
```

**判定规则**：
```javascript
function checkScopeCreep(roadmapContent) {
  // 逐行检查
  for (let line of lines) {
    const hasKeyword = scopeCreepKeywords.some(kw => line.includes(kw));
    if (hasKeyword) {
      // 检查前后 3 行是否有缓解措施
      const context = get_context(line, 3);
      const hasMitigation = mitigationKeywords.some(kw => context.includes(kw));
      if (!hasMitigation) {
        warnings.push(line);
      }
      // 有缓解措施 → 通过，不记录 warning
    }
  }
  return { pass: true, warning: warnings.length > 0 };
}
```

## 评分决策规则

### Critical 项检查

| 场景 | Critical 项（R0-R3） | 决策 |
|------|---------------------|------|
| 全部通过 | ✅ R0 ✅ R1 ✅ R2 ✅ R3 | 进入 Non-Critical 检查 |
| 任一失败 | ❌ 任一失败 | **驳回**（重新生成，最多 3 次） |

### Non-Critical 项检查

| 场景 | Non-Critical 项（R4 + 1-5, 7-8） | 决策 |
|------|--------------------------------|------|
| 全部通过 | ✅ 全部通过 | **通过**（10/10） |
| 1-2 项失败 | ⚠️ 1-2 项失败 | **条件通过**（修正后输出，8-9/10） |
| 3 项及以上失败 | ❌ 3 项及以上失败 | **驳回**（重新生成，<8/10） |

### SELF-REVIEW.md 生成逻辑

| 场景 | Critical 项 | Non-Critical 项 | 生成 SELF-REVIEW.md |
|------|------------|-----------------|---------------------|
| 全部通过 | ✅ 全部通过 | ✅ 全部通过 | ❌ **不生成** |
| 小问题 | ✅ 全部通过 | ⚠️ 1-2 项失败 | ❌ **不生成**（修正后直接输出） |
| Critical 失败 | ❌ 任一失败 | - | ✅ **生成**（记录失败原因用于调试） |

**优化理由**：
- 符合"不生成额外文件"原则
- 减少文件冗余
- Critical 项失败时才需要记录审阅过程用于调试

---

## 修正流程

### 评分 <10 分但 Critical 全部通过

1. **内存中修正**（不生成 SELF-REVIEW.md）
2. 直接输出修正后的 ROADMAP.md
3. 记录修正日志到流程引擎日志

### Critical 项失败

1. **生成 SELF-REVIEW.md**（记录失败原因）
2. 重新生成开发计划
3. 最多 3 次重试机会
