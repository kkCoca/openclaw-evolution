## 1. 产品概述

### 1.1 产品定位

URI to Markdown 是一个**通用型网页内容转换工具**，将任意网页 HTML 内容转换为结构化 Markdown 格式，支持登录认证、多 iframe 合并、表单字段提取等高级功能。

### 1.2 目标用户

| 用户类型 | 使用场景 | 核心需求 |
|---------|---------|---------|
| **知识管理者** | 知识沉淀、文档归档 | 批量转换、格式保留 |
| **技术支持工程师** | 致远 OA BUG 单处理 | 82 字段提取、附件列表 |
| **内容创作者** | 内容二次加工 | 格式 clean、易于编辑 |
| **开发者** | 集成到自动化流程 | CLI/API 调用、脚本化 |

### 1.3 核心价值

- ✅ **通用性** — 不绑定特定业务系统，支持任意网站
- ✅ **智能化** — 自动登录、多 iframe 合并、字段配对
- ✅ **多形态** — Chrome 插件 + CLI + 核心库 + HTTP API
- ✅ **可配置** — YAML 配置 + 环境变量，灵活适配

## 2. 功能需求

### 2.1 核心功能（P0）

#### 2.1.1 URI 转换

| 功能点 | 说明 | 验收标准 |
|-------|------|---------|
| HTML 转 Markdown | 使用 Turndown 引擎转换 | 保留标题/列表/代码块/链接/图片 |
| 内容提取 | 智能识别正文，过滤导航/广告 | 正文提取率 > 95% |
| 格式保留 | 保留原始格式结构 | 标题层级正确、列表完整 |

#### 2.1.2 登录认证

| 功能点 | 说明 | 验收标准 |
|-------|------|---------|
| 配置型认证 | YAML 配置站点认证信息 | 支持多站点配置 |
| 环境变量 | 支持 `${VAR}` 引用环境变量 | 密码不硬编码 |
| 自动登录 | 访问目标页前自动完成登录 | 登录成功率 > 99% |
| Cookie 管理 | 自动管理 Cookie 过期 | 过期自动重新登录 |

#### 2.1.3 多 iframe 合并

| 功能点 | 说明 | 验收标准 |
|-------|------|---------|
| iframe 检测 | 自动检测页面所有 iframe | 检测率 100% |
| 内容提取 | 提取每个 iframe 的完整内容 | 无内容丢失 |
| 智能合并 | 按逻辑顺序合并有内容的 iframe | Frame 2 头部 + Frame 4 详情 |

#### 2.1.4 致远 OA 特殊处理

| 功能点 | 说明 | 验收标准 |
|-------|------|---------|
| 字段提取 | 82 个字段，16 个区域 | 完整提取所有字段 |

| 表格提取 | 15 个表格完整提取 | 表格结构完整 |
| 附件处理 | 文件名 + 大小，URL 标注 | 附件列表完整 |

## 5. 用户故事

### 5.1 Chrome 插件场景

**故事 1：转换无需登录的网站**
```
作为 内容创作者
我希望 打开任意网页后一键转换为 Markdown
以便于 快速保存和二次编辑

验收标准:
- Given 打开 https://juejin.cn/post/123456
- When 点击 Chrome 插件图标
- Then 自动转换为 Markdown 并显示预览
- And 支持复制/下载
```

**故事 2：转换需要登录的内部网站**
```
作为 技术支持工程师
我希望 在致远 OA 中一键转换 BUG 单
以便于 快速归档和分享

验收标准:
- Given 已登录致远 OA
- When 打开 BUG 单详情页并点击插件
- Then 自动使用当前 Cookie 访问
- And 输出完整 Markdown（82 字段 + 附件列表）
```

### 5.2 CLI 场景

**故事 3：批量转换客户 BUG 单**
```
作为 技术支持工程师
我希望 批量转换多个客户 BUG 单
以便于 统一归档到知识库

验收标准:
- Given 准备了 urls.txt（包含 10 个 BUG 单 URL）
- When 执行 uri2md --batch urls.txt --output-dir ./bugs/
- Then 自动逐个转换并保存到指定目录
- And 显示处理进度和结果统计
```

**故事 4：指定配置文件**
```
作为 多账号用户
我希望 指定不同的配置文件
以便于 使用不同账号访问

验收标准:
- Given 配置了 config-account1.yaml 和 config-account2.yaml
- When 执行 uri2md --config config-account2.yaml <url>
- Then 使用 account2 的认证信息访问
```

### 5.3 HTTP API 场景

**故事 5：集成到智能运维系统**
```
作为 智能运维系统开发者
我希望 调用 HTTP API 转换页面
以便于 自动化知识沉淀

验收标准:
- Given 已启动 uri-to-markdown-api 服务
- When POST /api/convert {url, withMeta: true}
- Then 返回 Markdown 内容和元数据
- And 支持异步任务查询
```

## 6. 配置规范

### 6.1 config.yaml 结构

```yaml
# 全局配置
global:
  timeout: 30000              # 请求超时（毫秒）
  retryCount: 3               # 失败重试次数
  outputFormat: markdown+meta # 输出格式
  imageHandling: keep-url     # 图片处理方式

# 站点配置
sites:
  - domain: xt.seeyon.com
    loginUrl: https://xt.seeyon.com/seeyon/main.do
    username: ${SEEEYON_USER}
    password: ${SEEEYON_PASS}
    loginMethod: form

  - domain: juejin.cn
    # 无需认证

# Chrome 插件配置
chrome-extension:
  showMetaByDefault: true
  downloadPath: ~/Downloads
  shortcut: Alt+M
```

### 6.2 环境变量

```bash
# .env 文件
SEEEYON_USER=your_username
SEEEYON_PASS=your_password
```
